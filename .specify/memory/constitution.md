# Mediateca Broadcast Constitution

## Core Principles

### I. Rails 8 and Hotwire First
Все пользовательские сценарии MUST реализовываться на стандартном стеке Rails 8:
Server Rendering, Turbo, Stimulus, Active Record, Active Job и встроенные Rails 8
компоненты для фоновых задач, кеша и real-time коммуникаций. Добавление внешних
frontend-фреймворков допускается только при документированном обосновании
невозможности решить задачу средствами Hotwire. UI MUST строиться на Slim +
ViewComponent и **Tailwind CSS** для единообразия представлений, стилизации и
переиспользуемости интерфейса.
Inertia MAY использоваться точечно поверх Hotwire-подхода в экранах с высокой
интерактивностью, где это измеримо улучшает UX (скорость взаимодействия,
снижение когнитивной нагрузки, плавность сложных состояний).
Административная панель для внутренних операций MUST строиться на геме **Avo**
(Admin Panel Framework); в зоне маршрутов Avo допускается отход от Slim +
ViewComponent при условии, что бизнес-инварианты и правила домена не
дублируются в админке ad-hoc, а авторизация согласована с Pundit или CanCanCan.

### II. Domain-Driven Design by Bounded Contexts
Бизнес-логика MUST быть выражена через доменную модель и bounded contexts, а не
через контроллеры или ad-hoc сервисы. Инварианты домена MUST жить рядом с
агрегатами/сущностями. Application-слой координирует сценарии, но не содержит
правила домена. Интеграции с внешними системами MUST изолироваться через
адаптеры/порты.

### III. Test-Driven Development Is Non-Negotiable
Каждая функциональность MUST начинаться с теста (Red -> Green -> Refactor). Для
изменений в домене обязательны unit-тесты модели/доменного сервиса, для
пользовательских сценариев обязательны системные тесты (Capybara + Turbo flows),
для контрактов интеграций обязательны контрактные/интеграционные тесты. Код без
падающего теста до реализации считается нарушением конституции.

### IV. Explicit Architecture and Functional Decomposition
Каждый класс и метод MUST иметь одну четкую ответственность. Методы длиннее
одного логического шага SHOULD декомпозироваться на частные методы/объекты.
Дублирование MUST устраняться через рефакторинг. Контроллеры остаются тонкими:
валидация входа, вызов use-case, формирование HTTP-ответа. Непрозрачная логика
в callbacks и concerns MUST быть минимизирована и обоснована.

### V. Reliability, Observability and Secure Defaults
Все фоновые и внешние операции MUST быть идемпотентными и наблюдаемыми:
структурированные логи, трассировка ошибок, метрики времени и частоты сбоев.
Операции с данными MUST использовать транзакции там, где важна целостность.
Аутентификация, авторизация, CSRF-защита, параметризация запросов и защита
секретов MUST соблюдаться по умолчанию. Разграничение прав MUST выражаться через
централизованные политики (Pundit) или ability (CanCanCan), а не через разрозненные
условия в представлениях и контроллерах.

## Технологические стандарты Rails 8 + Hotwire

- Проект MUST использовать Ruby 4.x как основную runtime-версию.
- Rails MUST соответствовать актуальной стабильной версии проекта.
- Основная СУБД приложения (OLTP, Active Record) MUST быть **PostgreSQL 18**
  (одинаковая мажорная версия в Devcontainer, CI, staging и production); иная
  первичная база, другая мажорная версия или замена допускаются только с
  документированным ADR и планом миграции данных.
- Локальная разработка MUST вестись через Devcontainer с закрепленными версиями
  системных зависимостей и инструментов.
- Стиль кода MUST проходить RuboCop и соответствовать Ruby Style Guide.
- Авторизация (кто может выполнять действие или видеть данные) MUST
  реализовываться через **Pundit** или **CanCanCan** единообразно по всему
  приложению; одновременное использование обеих библиотек запрещено без
  документированного ADR и плана миграции. Для **Avo** MUST использоваться тот же
  механизм авторизации, что и для остального приложения (политики/abilities).
- Административный интерфейс MUST реализовываться с **Avo**; новые операции в
  админке оформляются через ресурсы/инструменты Avo с явными правилами доступа.
- Представления MUST писаться на Slim; ERB допускается только в legacy-коде.
- Стилизация frontend MUST выполняться через **Tailwind CSS**; добавление
  альтернативных CSS-фреймворков допускается только с ADR и планом миграции.
- UI-компоненты MUST реализовываться через ViewComponent с изолированными тестами.
- Новые UI-потоки MUST использовать Turbo Frames/Streams и progressive enhancement.
- Использование Inertia MUST сопровождаться кратким ADR/обоснованием в спецификации:
  почему Hotwire-реализация недостаточна и какие UX-метрики улучшаются.
- Stimulus-контроллеры MUST быть небольшими, переиспользуемыми и без хранения
  бизнес-логики.
- Миграции MUST быть обратимыми или сопровождаться явным планом отката.
- Критичные запросы к БД MUST проверяться на N+1 и покрываться индексами.
- Каждая публичная доменная операция MUST иметь краткую документацию использования.
- Production-деплой MUST выполняться через Kamal с единым описанным пайплайном
  сборки, выката и отката.

## Процесс разработки и Quality Gates

Для каждой задачи команда MUST соблюдать следующий поток:

1. Описать доменный сценарий и границы bounded context.
2. Написать тесты (unit/integration/system) и зафиксировать Red-состояние.
3. Реализовать минимальное решение до Green.
4. Выполнить рефакторинг с сохранением Green и улучшением читаемости.
5. Проверить соответствие RuboCop, тестам, архитектурным ограничениям и
   требованиям безопасности.

Минимальные quality gates для merge:

- ✅ Все тесты проходят локально и в CI.
- ✅ Нет критичных замечаний RuboCop и линтера безопасности.
- ✅ Изменения в домене отражены в тестах и документации.
- ✅ Для изменений схемы есть стратегия выката и отката.
- ✅ Изменения среды разработки синхронизированы с Devcontainer-конфигурацией.
- ✅ Изменения деплоя отражены в Kamal-конфигурации и проверены на staging.
- ✅ Новые или изменённые правила доступа отражены в политиках/abilities и
  покрыты тестами авторизации.
- ✅ Изменения в админке (Avo resources/actions) согласованы с авторизацией и не
  обходят доменные инварианты без явного обоснования.

Шаблон ADR для использования Inertia (MUST при каждом внедрении):

1. Контекст: экран/сценарий, целевой пользователь и текущая UX-проблема.
2. Почему не Hotwire-only: ограничения Turbo/Stimulus в данном кейсе.
3. Решение: границы применения Inertia и взаимодействие с Rails backend.
4. UX-метрики: что измеряем (время до действия, completion rate, error rate и т.д.).
5. План отката: как вернуться к Hotwire-only реализации при ухудшении метрик.

## Governance

Эта конституция имеет приоритет над локальными практиками команды. Любое
изменение принципов MUST оформляться как явная правка этого документа с
обоснованием, оценкой влияния на существующий код и планом миграции.

Политика версионирования конституции:

- MAJOR: несовместимые изменения принципов или удаление разделов.
- MINOR: добавление нового принципа/обязательного процесса.
- PATCH: уточнения формулировок без изменения смысла.

Проверка соответствия конституции MUST выполняться на этапе code review и перед
релизом. Нарушения MUST фиксироваться как технический долг с датой устранения.
