# Feature Specification: Медиатека и трансляция плейлистов в торговых точках

**Feature Branch**: `001-media-playlist-broadcast`  
**Created**: 2026-03-31  
**Status**: Draft  
**Input**: User description: "multi-tenant SaaS приложение для загрузки аудио и видео контента в личный кабинет пользователя, из загруженных файлов можно создавать плей-листы, эти плейлисты можно транслировать на устройства трансляции (большие телевизоры) установленные в различных торговых точках в разных городах, подразумевается что в основном это рекламные аудио и видео материалы, поддерживаемые форматы: видео (MP4 и WEBM), изображения (PNG, JPEG, GIF, WEBP), звук (MP3, WAV), презентации (PPT, PPTX), документы (PDF), точки для трансляции можно объединять в группы (например группа в рамках одного супермаркета). При просмотре загруженного файла показывается превью. Для загруженных файлов асинхронно определяется продолжительность и формат. Для выбора точек трансляции у каждой точки есть набор тегов, которые можно задавать при редактировании точки и затем по этим тегам можно фильтровать точки, набор тегов гибкий не фиксированный, можно добавлять произвольный тег. Можно создавать расписания трансляций для связки времени трансляции с плейлистом и группами устройств трансляции."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Загрузка медиа и превью в кабинете (Priority: P1)

Пользователь организации-заказчика открывает личный кабинет, загружает файл
поддерживаемого типа, видит статус обработки и превью; после полной обработки
видит определённые системой длительность (для подходящих типов) и уточнённый
формат содержимого.

**Why this priority**: без медиатеки нельзя собирать плейлисты и транслировать
контент — это фундамент продукта.

**Independent Test**: загрузка файла каждого поддерживаемого семейства типов,
проверка превью, отображения статуса анализа и итоговых метаданных у одного
пользователя в одной организации.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован в организации **When** он загружает файл
   допустимого типа **Then** файл появляется в библиотеке с превью и индикатором
   обработки, после завершения анализа отображаются формат и длительность (если
   применимо).
2. **Given** файл недопустимого типа или повреждён **When** пользователь
   пытается загрузить его **Then** система отклоняет загрузку с понятным
   сообщением.

---

### User Story 2 - Плейлисты из библиотеки (Priority: P2)

Пользователь создаёт плейлист, добавляет в него уже загруженные материалы,
меняет порядок элементов (в том числе перетаскиванием указателя мыши по списку —
drag and drop) и сохраняет; плейлист отображается в списке и открывается для
просмотра состава.

**Why this priority**: плейлист — единица, которую удобно назначать на эфир.

**Independent Test**: создание плейлиста только из уже существующих файлов,
изменение порядка перетаскиванием указателя мыши, сохранение, повторное открытие
и проверка того же порядка — без участия устройств и расписания.

**Acceptance Scenarios**:

1. **Given** в библиотеке есть минимум два файла **When** пользователь создаёт
   плейлист и добавляет их **Then** оба элемента отображаются в заданном порядке.
2. **Given** открыт плейлист **When** пользователь удаляет элемент из середины
   **Then** остальные элементы сохраняют корректный порядок.
3. **Given** в плейлисте не менее трёх элементов **When** пользователь перетаскивает
   один элемент в другую позицию указателем мыши и фиксирует изменения (сохраняет)
   **Then** новый порядок отображается в кабинете и используется при эфирном
   воспроизведении этого плейлиста.

---

### User Story 3 - Точки трансляции, теги и группы (Priority: P3)

Администратор (или роль с правом настройки) регистрирует точки трансляции
(устройства в торговых точках), задаёт каждой точке произвольные теги,
находит точки по фильтру тегов, объединяет точки в группу (например все экраны
одного супермаркета).

**Why this priority**: без модели точек и групп невозможно адресно назначать
эфир на сети магазинов.

**Independent Test**: создание нескольких точек с пересекающимся набором тегов,
фильтрация, создание группы и включение в неё подмножества точек — без загрузки
медиа и расписаний.

**Acceptance Scenarios**:

1. **Given** точка создана **When** пользователь добавляет новый произвольный
   тег **Then** тег доступен для поиска и фильтрации точек.
2. **Given** несколько точек **When** пользователь создаёт группу и включает в
   неё выбранные точки **Then** группа отображает ожидаемый состав.

---

### User Story 4 - Расписание трансляций (Priority: P4)

Пользователь создаёт правило расписания: какой плейлист и в какое время (или
интервал) должен идти на выбранные группы точек; может просматривать и изменять
расписание.

**Why this priority**: связывает контент с инфраструктурой эфира по времени.

**Independent Test**: создание расписания на существующий плейлист и существующую
группу с конкретным временным окном, затем изменение и проверка отображения.

**Acceptance Scenarios**:

1. **Given** есть плейлист и группа точек **When** пользователь создаёт запись
   расписания на диапазон времени **Then** система сохраняет связь плейлист —
   время — целевая группа.
2. **Given** два пересекающихся по времени назначения на одну и ту же группу
   **When** пользователь сохраняет второе **Then** система явно сигнализирует
   о конфликте и не оставляет состояние неоднозначным без подтверждения политики
   разрешения (см. Assumptions).

---

### User Story 5 - Воспроизведение на устройстве по расписанию (Priority: P5)

Устройство трансляции, привязанное к точке, получает назначание и воспроизводит
элементы плейлиста в соответствии с расписанием организации (визуально и/или
звук в зависимости от типа контента).

**Why this priority**: подтверждает ценность продукта для рекламного сценария в
точке продаж.

**Independent Test**: для тестовой точки и тестового расписания проверить, что
на экране отображается ожидаемое видео/изображение/звуковая дорожка согласно
очереди плейлиста в окне эфира.

**Acceptance Scenarios**:

1. **Given** активное расписание и онлайн-устройство на точке **When** наступает
   время эфира **Then** устройство начинает воспроизведение плейлиста в порядке
   элементов.
2. **Given** элемент плейлиста — статичное изображение с заданной длительностью
   показа **When** он выводится на экран **Then** изображение отображается в течение
   согласованного времени перед переходом к следующему элементу.

---

### Edge Cases

- Загрузка очень больших файлов: лимиты размера, прерывание сети, повтор попытки.
- Неполный или ошибочный анализ длительности/формата (например экзотический
  контейнер): отображение состояния ошибки и возможность повторной обработки.
- Устройство офлайн в момент начала эфира: поведение при появлении связи (догон
  или пропуск окон — см. Assumptions).
- Пересечение расписаний на одну группу или точку; смена часового пояса точки.
- Перетаскивание элемента плейлиста: отмена (отпускание вне допустимой зоны),
  прокрутка длинного списка во время перетаскивания, одновременное редактирование
  плейлиста двумя пользователями.
- PDF и офисные презентации: нет единого «плеера» на ТВ — требуется согласованная
  стратегия показа (см. Assumptions).
- Удаление файла, который входит в плейлист или активное расписание.
- Многопользовательское редактирование одной и той же сущности (плейлист,
  расписание).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST поддерживать мультитенантность: данные одной
  организации недоступны пользователям другой организации.
- **FR-002**: Пользователь MUST иметь доступ к личному кабинету организации с
  библиотекой загруженных материалов.
- **FR-003**: Система MUST принимать загрузку файлов следующих семейств типов:
  видео MP4 и WEBM; изображения PNG, JPEG, GIF, WEBP; звук MP3 и WAV; презентации
  PPT и PPTX; документы PDF.
- **FR-004**: Система MUST отображать превью для загруженного материала в списке и
  при детальном просмотре (для типов, где превью технически оправдано; для чистого
  звука — согласованное представление, например обложка/иконка, см. Assumptions).
- **FR-005**: Система MUST асинхронно определять технический формат и длительность
  контента там, где это применимо, и показывать статус обработки до завершения.
- **FR-006**: Пользователь MUST иметь возможность создавать плейлисты из файлов
  библиотеки, менять порядок элементов и сохранять изменения; смена порядка MUST
  поддерживаться перетаскиванием указателя мыши (drag and drop) в списке
  элементов плейлиста.
- **FR-007**: Пользователь MUST иметь возможность создавать и редактировать точки
  трансляции, включая привязку к месту (город, торговая точка по описанию
  пользователя).
- **FR-008**: У каждой точки MUST быть набор тегов; пользователь MAY добавлять
  произвольные новые теги; система MUST позволять фильтровать точки по одному или
  нескольким тегам.
- **FR-009**: Пользователь MUST иметь возможность объединять точки в группы и
  управлять составом группы.
- **FR-010**: Пользователь MUST иметь возможность создавать расписания, связывающие
  плейлист, временное окно и одну или несколько групп точек (или явный перечень
  точек — см. Assumptions).
- **FR-011**: Устройство трансляции MUST получать назначения для своей точки и
  воспроизводить контент плейлиста в рамках активного расписания.
- **FR-012**: Система MUST отклонять загрузки неподдерживаемых типов и сообщать
  причину пользователю.

### Architecture & Domain Requirements *(mandatory)*

- **ADR-001**: В спецификации и сопутствующей документации MUST быть зафиксированы
  границы домена «Медиатека и каталог», «Эфир и расписание» и «Устройства и точки»
  с явными бизнес-инвариантами (например «элемент плейлиста ссылается только на
  актив в своей организации»).
- **ADR-002**: Правила предметной области (who can broadcast what, когда
  допустимо удаление, как разрешаются конфликты расписаний) MUST быть определены
  явно и проверяемы, а не скрыты только в интерфейсной логике.
- **ADR-003**: Любые будущие интеграции (облачное хранилище, внешние каталоги,
  аналитика показов) MUST описываться как отдельные контракты без смешения с
  ядром медиатеки и расписания.

### Frontend Strategy Requirements *(mandatory for UI features)*

- **UIR-001**: Интерфейс личного кабинета MUST обеспечивать быструю работу со
  списками (медиа, точки, расписание) без потери контекста при типовых действиях.
- **UIR-002**: Повторяющиеся блоки интерфейса (карточка файла, строка точки, блок
  расписания) MUST отображаться единообразно во всём продукте; при изменении
  порядка в плейлисте перетаскиванием пользователь MUST видеть понятную обратную
  связь (целевая позиция/подсветка) до сохранения.
- **UIR-003**: Для экранов с исключительно высокой интерактивностью (если
  признаются таковы на этапе проекта) MUST быть зафиксированы целевые
  пользовательские метрики, объём внедрения и план отката к более простому
  варианту взаимодействия.

### Key Entities *(include if feature involves data)*

- **Организация (тенант)**: изолированный клиент SaaS; владеет пользователями,
  медиа, точками и расписаниями.
- **Пользователь**: учётная запись с ролями внутри организации (детали ролей —
  вне минимума спецификации, см. Assumptions).
- **Медиафайл (актив)**: загруженный объект с типом, превью, результатами анализа
  длительности и формата, статусом обработки.
- **Плейлист**: упорядоченный список ссылок на медиафайлы организации.
- **Точка трансляции**: логическое устройство в торговой точке, набор тегов,
  привязка к группам, состояние доступности для эфира.
- **Группа точек**: именованная группа для адресации эфира.
- **Тег**: произвольная метка для фильтрации точек; создаётся при использовании.
- **Расписание**: правило «когда и что играть» с временным окном и целями
  (группы/точки); может ссылаться на один плейлист за окно (см. Assumptions).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Не менее 95% успешных загрузок поддерживаемых типов завершают этап
  анализа формата и длительности без ручного вмешательства персонала поддержки в
  течение одной рабочей смены.
- **SC-002**: Пользователь создаёт плейлист из 10 элементов, корректирует порядок
  в том числе перетаскиванием, и сохраняет итог за не более чем 3 минуты при
  знакомстве с интерфейсом (по результатам пользовательского тестирования на
  выборке ≥ 5 операторов).
- **SC-003**: Фильтрация списка из 200 точек по комбинации тегов возвращает
  результат для пользователя не более чем за 2 секунды воспринимаемого ожидания
  в типовых условиях офисной сети.
- **SC-004**: В пилоте на ≤ 50 устройствах не менее 98% запланированных окон эфира
  начинаются не позднее чем через 60 секунд от объявленного времени начала.

### Engineering Quality Outcomes *(mandatory)*

- **EQ-001**: Для каждой пользовательской истории заранее описаны приёмочные
  сценарии и критерии «готово», до начала поставки функциональности.
- **EQ-002**: Бизнес-правила удаления медиа, правки плейлиста и конфликтов
  расписания воспроизводимы в виде проверяемых сценариев.
- **EQ-003**: Сквозные сценарии «загрузка → плейлист → расписание → эфир на
  точке» покрыты сквозной проверкой до выпуска пилота.

## Assumptions

- Организация — основной арендатор; внутри одной организации несколько
  пользователей с общей библиотекой (деталь матрицы ролей уточняется при плане
  доступа).
- Расписание хранится в универсальном времени с привязкой отображения к часовому
  поясу точки трансляции; если у точки пояс не задан, используется пояс
  организации.
- Конфликт пересекающихся активных расписаний на одну цель по умолчанию запрещён;
  альтернативная политика «приоритет правила» может быть введена отдельным
  решением до реализации FR-010 второго сценария приёмки.
- Для PDF и презентаций PPT/PPTX система преобразует контент в форму,
  гарантированно воспроизводимую на целевом экране (например последовательность
  страниц как изображения или видео-рендер), если заказчик не задаёт иной
  стандарт на этапе плана.
- Устройство в состоянии офлайн получает пропущенный эфир согласно политике:
  по умолчанию — начать текущий плейлист при восстановлении связи без
  воспроизведения полностью прошедших окон (уточним при проектировании протокола
  устройства).
- Лимиты размера файла и квоты хранения задаются политикой продукта и
  тарификацией (конкретные числа — вне рамок данной спецификации).
