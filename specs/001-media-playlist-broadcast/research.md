# Research: медиатека и трансляция в торговых точках

**Feature**: `001-media-playlist-broadcast`  
**Date**: 2026-03-31

## 1. Авторизация: Pundit vs CanCanCan

**Decision**: **Pundit** как единый механизм политик для приложения и **Avo**
(`authorize @record` / `avo_authorize`).

**Rationale**: Политики на класс + запись хорошо ложатся на мультитенантные
проверки (`record.organization_id == Current.organization_id`), проще
покрывать тестами на уровне `Policy`, документация Avo описывает интеграцию с
Pundit.

**Alternatives considered**: CanCanCan — компактные `Ability`, но проект уже
ориентирован на явные Policy-объекты в конституции; выбор возможен при ADR, если
команда унифицирует на Ability.

---

## 2. Хранение файлов и превью

**Decision**: **Active Storage** (PostgreSQL + диск или S3-совместимое объектное
хранилище в production); превью/варианты — **Active Storage previews** для
изображений и **отдельный пайплайн обработки** (см. п. 3) для видео/аудио
метаданных и производных превью где нужно.

**Rationale**: Стандарт Rails 8, совместим с Kamal и blob-моделью;
мультитенантность обеспечивается связью `MediaAsset`/`Organization` и политиками,
а не публичными префиксами без проверки.

**Alternatives considered**: Прямая работа только с объектным хранилищем без
Active Storage — больше кода и риск расхождения метаданных с файлами.

---

## 3. Асинхронный анализ формата и длительности

**Decision**: **Active Job** + **Solid Queue** (или Sidekiq, если уже в проекте)
для задачи `ProcessMediaMetadataJob`; для видео/аудио — **ffprobe** (через
обёртку в контейнере) для `duration` и `codec/container`; для PDF/PPT — отдельные
шаги конвертации (см. п. 4) с идемпотентными job по `blob_id` / `media_asset_id`.

**Rationale**: Соответствует конституции (фоновые задачи идемпотентны и
логируемы); ffprobe — индустриальный стандарт для метаданных.

**Alternatives considered**: Только клиентские метаданные — недостаточно для
качества и безопасности на сервере.

---

## 4. PDF и PPTX на экране (согласно допущению спецификации)

**Decision**: При загрузке PDF/PPTX — **фоновая конвертация** в последовательность
**изображений** (или один прокручиваемый «слайд-ролл» как набор файлов),
сохранённых как производные blob / отдельные `PlaylistItem` типа «слайд». Плеер
на устройстве воспроизводит уже **растровые кадры** или **готовое видео**,
сгенерированное из слайдов (выбор уточняется в MVP: сначала изображения + таймер
показа, проще для спецификации US5).

**Rationale**: Соотносится с Assumptions в `spec.md`; ТВ-браузеры надёжнее
показывают изображения/MP4, чем нативный PPTX.

**Alternatives considered**: Онлайн-рендер PPT на устройстве — высокий риск и
зависимость от клиента.

---

## 5. Протокол устройства и получение расписания

**Decision**: **HTTPS JSON API** для плеера: регистрация/паринг `device_token`,
`GET /api/v1/devices/me/schedule` (или long polling) + **`GET` подписанных URL**
на медиа для воспроизведения; опционально **Turbo Streams / Action Cable** для
«толкания» обновлений расписания (второй этап). MVP: **опрос с интервалом**
(30–60 с) + принудительный `sync` по действию в кабинете.

**Rationale**: Проще отлаживать на «больших ТВ» с WebView; контракт явно
версионируемый (`/api/v1/`).

**Alternatives considered**: Только WebSocket без polling — сильнее зависимость
от сети магазина; допустимо как эволюция.

---

## 6. Drag-and-drop порядка плейлиста

**Decision**: **SortableJS** (или нативный HTML5 DnD) через **Stimulus**
контроллер; порядок передаётся на сервер **одним запросом** (например `PATCH`
с массивом `playlist_item_ids`) с оптимистичным UI и откатом при ошибке.

**Rationale**: Соответствует Hotwire-first; без Inertia ADR.

---

## 7. Часовые пояса и расписание

**Decision**: Хранить `starts_at` / `ends_at` в **UTC**; у **Organization** и
**BroadcastPoint** — `time_zone` (строка TZDB); вычисление «локального окна» и
проверка пересечений — в доменном сервисе / query object; дублирование на одну
цель — **ошибка валидации** по умолчанию (как в спецификации).

**Rationale**: Совместимо с Assumptions `spec.md`.

---

## 8. Тестовый стек

**Decision**: **RSpec** + **FactoryBot**, системные спеки **Capybara + Cuprite**
или **Selenium** (в CI headless); контракты API устройств — **request specs** +
примеры JSON в `contracts/`.

**Rationale**: Соответствует конституции (system tests для Turbo).

**Alternatives considered**: Minitest — допустимо по ADR; по умолчанию для плана —
RSpec для единообразия с типичным Rails 8 коммерческим стеком.

---

## Итог

Все открытые из `Technical Context` пункты закрыты осознанными решениями;
отклонений от конституции нет. При смене Pundit на CanCanCan или отказе от
ffprobe требуется отдельный ADR.
